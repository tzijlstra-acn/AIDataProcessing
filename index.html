About This Blueprint

This implementation guide defines everything required to design, build, and deploy an AI-powered Internal Control System (ICS) Cockpit for a financial services institution. It covers the complete team structure, phased delivery roadmap, data ingestion requirements, AI/ML architecture, and feature-level interaction specification. This document is the contract between Accenture delivery and the client — every team member, every data source, and every user-facing feature described here must be agreed, provisioned, or delivered before the corresponding phase exit gate is passed.

Roadmap & Gantt

Six-phase, 28-week delivery plan with phase gates, Gantt timeline, risk register, and governance cadence.

Delivery Phases
Phase 0 Discovery & Setup Weeks 1–4

Establish project foundation — understand GRC landscape, validate data, agree architecture, and provision the development environment.

Key Activities
  • Week 1: Project kickoff; GRC system assessment; live demo of GRC platform API
  • Week 2: Data audit (extract 100–500 records per entity); TPRM tool assessment; Azure AD configuration review
  • Week 3: Draft Architecture Decision Record (ADR); DPIA draft with DPO
  • Week 4: Terraform apply (provision all Azure resources non-prod); CI/CD pipeline; Project Charter finalised
Deliverables
Data Readiness Report (DE+GRC, W4) ADR (SA, W3) Terraform IaC (SA, W4) Project Charter (PM, W4) DPIA draft (Accenture+DPO, W4)
Exit Criteria
  • Data Readiness Report accepted by GRC Domain Expert and client Risk SME
  • ADR signed off by client IT Lead
  • Dev environment access provisioned for all Accenture team members
  • Project Charter signed by Sponsor
Phase 1 Data Foundation Weeks 5–12

Build ingestion pipelines, database schema, and document pipeline; load historical data; produce static dashboard for design validation.

Key Activities
  • Weeks 5–6: Deploy PostgreSQL schema; build GRC connector; implement watermark-based incremental ingestion; initial data load
  • Weeks 7–8: ADF pipelines for remaining entities; dbt transformation layer; TPRM connector
  • Weeks 9–10: Document ingestion pipeline (blob→parse→chunk→embed→Azure AI Search)
  • Weeks 11–12: Static dashboard build (hardcoded, matching real client data structure); client design validation
Deliverables
Working GRC pipelines (DE, W10) Data quality report (DE+GRC, W10) Document ingestion pipeline (DE+AI, W11) Static dashboard (FE, W11) Client design sign-off (PM+GRC, W12)
Exit Criteria
  • All Critical entities loaded and quality-validated
  • Static dashboard reviewed and signed off by client
  • ADF pipelines running clean for 5 consecutive days
Phase 2 Core Dashboard Weeks 9–16

Connect dashboard to live API; deliver fully functional read-only dashboard with real data but no AI features.

Key Activities
  • Weeks 9–10: FastAPI skeleton; Azure AD auth middleware; RBAC; GET /api/overview, /api/areas, /api/processes; Redis caching
  • Weeks 11–12: Connect frontend to live API; KPI cards, area cards, risk heatmap (inherent/residual toggle)
  • Weeks 13–14: Risk register and controls with full filter support; process detail view; RCSA timeline view
  • Weeks 15–16: Cross-area risk view; performance testing (<500ms under 50 concurrent users); WCAG 2.1 AA review
Exit Criteria
  • Dashboard v1.0 demo'd and accepted by client
  • API data validated by GRC Domain Expert
  • Performance targets met (<500ms p95 under 50 concurrent users)
Phase 3 AI Layer Weeks 13–20

Add LLM agent chat, TPRM signal engine, trend detection, and RCSA due-date monitoring.

Key Activities
  • Weeks 13–14: Azure OpenAI deployment; LangChain agent with tool use; POST /api/agent/chat with SSE streaming
  • Weeks 15–16: System prompt engineering with GRC Domain Expert; context injection; agent audit logging
  • Weeks 17–18: TPRM Signal Engine (Service Bus subscriber); signal review UI; accept/reject workflow
  • Weeks 19–20: Trend detection (weekly Azure Function, scipy linear regression); RCSA due-date monitor (daily 07:00 UTC)
Exit Criteria
  • AI agent answers ≥80% of defined test query set correctly
  • TPRM engine generates draft risk from test event
  • DPIA signed by DPO
Phase 4 Workflow & Actions Weeks 17–24

Add notification infrastructure, 1LoD task engine, RCSA lifecycle orchestration, risk update workflow, and full RBAC enforcement.

Key Activities
  • Weeks 17–18: Azure Communication Services Email; Teams webhook; workflow_tasks management service
  • Weeks 19–20: Azure Durable Functions — RCSA Assessment lifecycle (D-14 reminder, D-3 escalation)
  • Weeks 21–22: PUT /api/risks/{'{id}'}; risk update form; 2LoD review and comment workflow
  • Weeks 23–24: Full RBAC enforcement; role-based UI rendering; end-to-end workflow test
Exit Criteria
  • Full RCSA lifecycle tested E2E in staging environment
  • RBAC validated for all four roles
  • Notification delivery confirmed (email + Teams)
Phase 5 Hardening & Rollout Weeks 21–28

Security hardening, performance tuning, UAT, user training, and production go-live.

Key Activities
  • Weeks 21–23: Internal security review + external pen test; remediate Critical/High findings
  • Weeks 22–24: Load testing (100 concurrent users); optimise indexes/Redis TTLs/Container Apps replicas
  • Weeks 24–26: UAT with 5–10 2LoD and 5–10 1LoD participants; Critical/High defects fixed within 48h
  • Weeks 25–28: User training (recorded videos); admin runbook; production provisioning; go-live + 2-week hypercare
Exit Criteria
  • All pen test Critical/High findings remediated
  • UAT sign-off obtained from client Risk Management Lead and PM
  • Production environment verified and go-live approved by Sponsor

Data Eng (GRC Connector)
Registered OAuth client in ServiceNow. Designed the incremental pull query using sys_updated_on watermark to minimise API calls and respect ServiceNow rate limits (600 req/hr).
Python GRCConnector — ServiceNowAdapter.fetch_risks()
# connectors/servicenow_adapter.py
class ServiceNowAdapter(GRCConnector):
    def fetch_risks(self, since: datetime) -> list[RiskRecord]:
        resp = self._session.get(
            f"{self.base_url}/api/now/table/sn_risk_risk",
            params={
                "sysparm_query": f"sys_updated_on>{since.isoformat()}",
                "sysparm_fields": RISK_FIELDS,
                "sysparm_limit": 500,
            },
        )
        resp.raise_for_status()
        raw = resp.json()["result"]

        return [
            RiskRecord(
                source_id=r["number"],
                title=r["short_description"],
                likelihood=int(r["likelihood"]),
                impact=int(r["impact"]),
                state=r["state"],
                owner_email=r["owner"],
                category=r["category"],
                process_id=r.get("u_process_id"),
                source_updated_at=parse_dt(r["sys_updated_on"]),
                raw=r,
            )
            for r in raw
        ]
# RSK-2024-0142 → RiskRecord(source_id="RSK-2024-0142",
#   likelihood=4, impact=4, process_id="FRP-002", …)
GRCConnector / ServiceNowAdapter (Python, Azure Function)
Runs inside ADF-triggered Python Azure Function · Parses ~120 records in <50ms · Retry: 3× exponential backoff
Accenture
Data Eng (GRC Connector)
The GRCConnector abstract class defines fetch_risks(). ServiceNowAdapter implements it for ServiceNow. A second adapter for Archer is being planned. Pydantic RiskRecord validates types and rejects malformed records into a dead-letter log.
Why an Adapter pattern?
Isolates GRC-tool specifics from pipeline logic. Swapping ServiceNow for Archer requires only a new Adapter — no pipeline changes.

Azure Data Factory — pl_grc_risks_incremental
Trigger: every 15 min · Total run: 1.64s · Watermark stored in Azure SQL control table · Cost: ~€0.001/run
Accenture
Data Eng (ADF)
Pipeline uses a watermark pattern — reads the last successful run timestamp from a control table, fetches only records updated since then, and updates the watermark on success. Failed runs do not advance the watermark, so no records are skipped.
Client
IT / Azure Lead

PostgreSQL 16 — staging.risks_raw
JSONB raw payload preserved for replay · ON CONFLICT UPSERT is idempotent · <10ms per row
Accenture
Data Eng (schema design)
Staging schema stores the raw JSONB payload alongside the pipeline run ID. This makes reprocessing after connector bugs trivial — re-run dbt against the existing staging rows. The staging table is the immutable source of truth for each extraction run.
Data Quality Warning
If u_process_id is NULL (legacy ServiceNow records), the record still lands in staging but dbt will set process_id = NULL and the risk will appear in the heatmap with no process linkage.
dbt model — models/core/risks.sql
-- models/core/risks.sql
{{ config(materialized='incremental',
          unique_key='source_id',
          on_schema_change='append_new_columns') }}

WITH src AS (
  SELECT
    source_id,
    raw_payload ->> 'short_description'  AS title,
    (raw_payload ->> 'likelihood')::int  AS likelihood,
    (raw_payload ->> 'impact')::int      AS impact,
    raw_payload ->> 'state'             AS status,
    raw_payload ->> 'owner'             AS owner_email,
    raw_payload ->> 'category'          AS category,
    raw_payload ->> 'u_process_id'      AS process_id,
    fetched_at
  FROM staging.risks_raw
  WHERE source_system = 'servicenow'
    {% if is_incremental() %}
    AND fetched_at > (SELECT MAX(dbt_updated_at) FROM {{ this }})
    {% endif %}
)
SELECT *, NOW() AS dbt_updated_at
FROM   src
dbt Core — models/core/risks.sql (incremental)
Runs after ADF completes via ADF-dbt activity · ~2.1s for full table · incremental: ~0.4s · dbt tests: not_null, unique, accepted_values
Accenture
Data Eng (dbt)
The incremental model only processes rows newer than the last dbt run, keeping transform times sub-second. dbt tests (not_null on source_id, unique on source_id, accepted_values on status) block promotion if data quality fails.

FastAPI — GET /api/heatmap?mode=inherent&scope=all
# routers/heatmap.py
@router.get("/heatmap")
async def get_heatmap(mode: str = "inherent",
                      scope: str = "all",
                      db: AsyncSession = Depends(get_db),
                      cache: Redis = Depends(get_redis)):
    cache_key = f"heatmap:{mode}:{scope}"
    cached = await cache.get(cache_key)
    if cached:
        return JSONResponse(json.loads(cached))

    rows = await db.execute(
        """SELECT likelihood, impact,
                  COUNT(*)            AS count,
                  MAX(rating)         AS max_rating,
                  ARRAY_AGG(source_id) AS risk_ids
           FROM core.risks
           WHERE status = 'Open'
           GROUP BY likelihood, impact"""
    )
    matrix = build_matrix(rows)
    await cache.setex(cache_key, 300, json.dumps(matrix))
    return matrix
# RSK-2024-0142 lands in cell {"likelihood":4,"impact":4}

01
Data Sovereignty
All compute and storage within client's Azure tenant in EU region. No data transmitted to external APIs. Azure OpenAI within the client tenant is the only LLM option.
02

Ingestion layer decoupled from storage, decoupled from AI layer. Each layer can be updated independently without impacting other layers.
03
Event-Driven Architecture
TPRM signals and RCSA status changes trigger downstream processing without polling. Azure Service Bus decouples producers and consumers.

Trend Detection Algorithm
Schedule: Azure Function weekly (Sunday 03:00 UTC) Algorithm: scipy.stats.linregress Min data points: 3 quarters LLM cost: €0 — pure Python
def detect_trends(risk_id: str, lookback_quarters: int = 4) -> TrendResult: history = risk_rating_history_repo.get(risk_id, lookback_quarters) if len(history) < 3: return TrendResult(risk_id=risk_id, trend='Insufficient data') scores = [h.residual_likelihood * h.residual_impact for h in history] slope, _, _, p_value, _ = linregress(range(len(scores)), scores) if p_value > 0.1: trend = 'Stable' elif slope > 0.5: trend = 'Increasing' elif slope < -0.5: trend = 'Decreasing' else: trend = 'Stable' return TrendResult(risk_id=risk_id, trend=trend, slope=slope)

Why Azure OpenAI (not Anthropic API direct)

  1. Data sovereignty: All processing stays within the client's Azure tenant. Anthropic's API is a public endpoint — transmitting client GRC data to it fails DPIA requirements for most EU financial institutions.
  2. Private Endpoints: Azure OpenAI can be locked behind a Private Endpoint with no public internet exposure.
  3. Compliance: EU data residency guaranteed (Sweden/Netherlands Azure regions).
  4. Existing commercial relationship: Most financial institutions already have Azure EA contracts covering Azure OpenAI consumption.

Context Optimisation Strategies

  1. Context pruning: Only inject the top 8 risks relevant to the current view. Saves ~4,000 tokens vs. injecting all 300+ risks.
  2. Conversation summarisation: After 10 turns, summarise with GPT-4o-mini (cheap), then continue with summary instead of raw turns.
  3. Redis cache for repeated queries: Cache agent responses by (query_hash + user_role + date) with TTL=1 hour. Potential 30–40% token reduction.
  4. Model downgrade for classification: GPT-4o-mini for all intent pre-classification — 5× cheaper input, 16× cheaper output.
