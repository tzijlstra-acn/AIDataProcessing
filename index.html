About This Blueprint

This implementation guide defines everything required to design, build, and deploy an AI-powered Internal Control System (ICS) Cockpit for a financial services institution. It covers the complete team structure, phased delivery roadmap, data ingestion requirements, AI/ML architecture, and feature-level interaction specification. This document is the contract between Accenture delivery and the client — every team member, every data source, and every user-facing feature described here must be agreed, provisioned, or delivered before the corresponding phase exit gate is passed.

Section 3

Roadmap

Six-phase, 28-week delivery plan with phase gates, Gantt timeline, and governance cadence.

3.1 Delivery Phases
Phase 0 Discovery & Setup Weeks 1–4

Establish project foundation — understand GRC landscape, validate data, agree architecture, and provision the development environment.

Entry Criteria
  • Signed engagement letter
  • PM and SA available from project start
  • Client Project Sponsor accessible for kickoff
Key Activities
  • Week 1: Project kickoff; GRC system assessment; live demo of GRC platform API
  • Week 2: Data audit (extract 100–500 records per entity); TPRM tool assessment; Azure AD configuration review
  • Week 3: Draft Architecture Decision Record (ADR); DPIA draft with DPO
  • Week 4: Terraform apply (provision all Azure resources non-prod); CI/CD pipeline; Project Charter finalised
Deliverables
Data Readiness Report (DE+GRC, W4) ADR (SA, W3) Terraform IaC (SA, W4) Project Charter (PM, W4) DPIA draft (Accenture+DPO, W4)
Exit Criteria
  • Data Readiness Report accepted by GRC Domain Expert and client Risk SME
  • ADR signed off by client IT Lead
  • Dev environment access provisioned for all Accenture team members
  • Project Charter signed by Sponsor
Phase 1 Data Foundation Weeks 5–12

Build ingestion pipelines, database schema, and document pipeline; load historical data; produce static dashboard for design validation.

Entry Criteria
  • Phase 0 exit criteria met
  • ≥60% of Critical items in Data Readiness Checklist received
  • GRC sandbox credentials issued
Key Activities
  • Weeks 5–6: Deploy PostgreSQL schema; build GRC connector; implement watermark-based incremental ingestion; initial data load
  • Weeks 7–8: ADF pipelines for remaining entities; dbt transformation layer; TPRM connector
  • Weeks 9–10: Document ingestion pipeline (blob→parse→chunk→embed→Azure AI Search)
  • Weeks 11–12: Static dashboard build (hardcoded, matching real client data structure); client design validation
Deliverables
Working GRC pipelines (DE, W10) Data quality report (DE+GRC, W10) Document ingestion pipeline (DE+AI, W11) Static dashboard (FE, W11) Client design sign-off (PM+GRC, W12)
Exit Criteria
  • All Critical entities loaded and quality-validated
  • Static dashboard reviewed and signed off by client
  • ADF pipelines running clean for 5 consecutive days
Phase 2 Core Dashboard Weeks 9–16

Connect dashboard to live API; deliver fully functional read-only dashboard with real data but no AI features.

Entry Criteria
  • PostgreSQL schema deployed and seeded
  • Static dashboard design approved by client
Key Activities
  • Weeks 9–10: FastAPI skeleton; Azure AD auth middleware; RBAC; GET /api/overview, /api/areas, /api/processes; Redis caching
  • Weeks 11–12: Connect frontend to live API; KPI cards, area cards, risk heatmap (inherent/residual toggle)
  • Weeks 13–14: Risk register and controls with full filter support; process detail view; RCSA timeline view
  • Weeks 15–16: Cross-area risk view; performance testing (<500ms under 50 concurrent users); WCAG 2.1 AA review
Deliverables
Live dashboard v1.0 (BE+FE, W16) All core API endpoints tested (BE, W16) Performance test report (BE+DevOps, W16)
Exit Criteria
  • Dashboard v1.0 demo'd and accepted by client
  • API data validated by GRC Domain Expert
  • Performance targets met (<500ms p95 under 50 concurrent users)
Phase 3 AI Layer Weeks 13–20

Add LLM agent chat, TPRM signal engine, trend detection, and RCSA due-date monitoring.

Entry Criteria
  • Document ingestion pipeline producing quality search results
  • ≥1 area's risk data fully loaded and validated
Key Activities
  • Weeks 13–14: Azure OpenAI deployment; LangChain agent with tool use; POST /api/agent/chat with SSE streaming; agent chat panel in frontend
  • Weeks 15–16: System prompt engineering with GRC Domain Expert; context injection; agent audit logging
  • Weeks 17–18: TPRM Signal Engine (Service Bus subscriber); signal review UI; accept/reject workflow
  • Weeks 19–20: Trend detection (weekly Azure Function, scipy linear regression); RCSA due-date monitor (daily 07:00 UTC)
Deliverables
Dashboard v2.0 with AI agent (BE+FE+AI, W18) TPRM signal engine (AI, W19) Trend detection (AI, W20) Agent quality evaluation report (AI+GRC, W20) DPIA finalised (DPO+Accenture, W18)
Exit Criteria
  • AI agent answers ≥80% of defined test query set correctly
  • TPRM engine generates draft risk from test event
  • DPIA signed by DPO
Phase 4 Workflow & Actions Weeks 17–24

Add notification infrastructure, 1LoD task engine, RCSA lifecycle orchestration, risk update workflow, and full RBAC enforcement.

Entry Criteria
  • Dashboard v2.0 accepted by client
  • Azure Communication Services provisioned
Key Activities
  • Weeks 17–18: Azure Communication Services Email; Teams webhook; workflow_tasks management service; reminder/escalate API
  • Weeks 19–20: Azure Durable Functions — RCSA Assessment lifecycle (D-14 reminder, D-3 escalation)
  • Weeks 21–22: PUT /api/risks/{'{id}'}; risk update form; 2LoD review and comment workflow
  • Weeks 23–24: Full RBAC enforcement; role-based UI rendering; end-to-end workflow test
Deliverables
Dashboard v3.0 full interactive (all teams, W24) Notification system (BE, W22) RCSA lifecycle workflow (BE, W22) RBAC enforced (BE+SA, W24) E2E integration test report (W24)
Exit Criteria
  • Full RCSA lifecycle tested E2E in staging environment
  • RBAC validated for all four roles
  • Notification delivery confirmed (email + Teams)
Phase 5 Hardening & Rollout Weeks 21–28

Security hardening, performance tuning, UAT, user training, and production go-live.

Entry Criteria
  • Dashboard v3.0 functional in staging environment
  • All Phase 4 exit criteria met
Key Activities
  • Weeks 21–23: Internal security review + external pen test; remediate Critical/High findings; Azure Defender posture review
  • Weeks 22–24: Load testing (100 concurrent users, peak RCSA period); optimise indexes/Redis TTLs/Container Apps replicas
  • Weeks 24–26: UAT with 5–10 2LoD and 5–10 1LoD participants; daily defect triage; Critical/High defects fixed within 48h
  • Weeks 25–28: User training (recorded videos); admin runbook; production environment provisioning; go-live + 2-week hypercare
Deliverables
Security review + pen test remediation (SEC, W23) Performance test report + prod sizing (BE+DevOps, W24) UAT sign-off (Client RM Lead+PM, W26) Training materials (CM, W27) Admin runbooks (SA+BE, W27) Production go-live (all teams, W28)
Exit Criteria
  • All pen test Critical/High findings remediated
  • UAT sign-off obtained from client Risk Management Lead and PM
  • Production environment verified and go-live approved by Sponsor

Section 4

Data Requirements

Every data source, entity, field, and access requirement needed before any feature can be built.

Prerequisites Warning

No AI component, no dashboard feature, and no workflow can be built without first securing the data described here. Missing or poor-quality data is the single most common cause of timeline slippage on GRC modernisation engagements. The 30-item Data Readiness Checklist (Section 4.5) must be completed and accepted in Phase 0 before Phase 1 commences.

4.1 GRC System Data

Risk Register — Most Critical Entity

All mandatory fields must be populated for Open risks. The risk register is the foundation of the dashboard — every heatmap cell, KPI metric, and AI agent answer depends on it.

Data Quality Criteria: All mandatory fields 100% populated for Open risks; inherent/residual rating fields ≥95% populated; historical records covering ≥2 complete RCSA cycles (typically 4+ quarterly snapshots).

Control Register

control_id, control_title, control_type (Preventive/Detective/Corrective), control_category (Manual/Automated/IT-Dependent Manual), process_id, control_owner, effectiveness_rating (Effective/Partially Effective/Ineffective/Not Assessed), last_test_date, next_test_date, test_frequency, key_control_indicator (Boolean), status

RCSA Assessment Results

assessment_id, assessment_cycle (Q1-2025 etc.), business_area, process_id, assessment_status (Not Started/In Progress/1LoD Complete/2LoD Review/Completed/Overdue), start_date, due_date, assigned_to, reviewer, overall_risk_rating, findings_count, open_actions_count

Process Inventory

process_id, process_name, business_area, process_owner, lod_designation (1LoD/2LoD), process_criticality, regulatory_relevance (DORA/GDPR/CRR etc.), rcsa_frequency, next_rcsa_due

Historical Risk Rating Snapshots

Quarterly/semi-annual snapshots going back ≥2 years. Fields: risk_id, snapshot_date, inherent_likelihood, inherent_impact, residual_likelihood, residual_impact. Required for trend detection algorithm.

Control Test Results

test_id, control_id, test_date, test_result (Pass/Fail/Pass with Exceptions), exceptions_noted, evidence_reference

Assessment Schedules (18-month forward)

schedule_id, process_id, cycle_name, due_date, assigned_to, type. Required for the RCSA timeline view and due-date monitor.

4.2 TPRM Data

Vendor Registry (Daily Refresh Minimum)

Key fields: vendor_id, vendor_name, vendor_classification (Critical/Important/Standard/Registered), dora_criticality, ict_service_flag, sub_processor_flag, contract_expiry_date, relationship_owner, business_areas_served, concentration_risk, last_assessment_date, next_assessment_due.

Trigger fields — change events required when ANY of these change: vendor_classification, ict_service_flag, dora_criticality, sub_processor_flag, status.

TPRM Change Event Schema (Service Bus Message)

{ "event_id": "evt-2025-0892", "event_type": "VENDOR_CLASSIFICATION_CHANGED", "vendor_id": "VND-0234", "vendor_name": "Acme Cloud Services Ltd", "previous_value": "Standard", "new_value": "Critical", "changed_field": "vendor_classification", "changed_at": "2025-07-14T10:32:00Z", "changed_by": "user@client.com" }

DORA ICT Third-Party Register

Required if client is an EU financial entity subject to DORA. Fields: ict_tpp_name, ict_tpp_lei (Legal Entity Identifier), service_type, data_classification, substitutability (Easy/Difficult/Impossible), jurisdiction_of_law, service_start_date.

4.3 Organisational Data

Org Hierarchy

org_unit_id, org_unit_name, org_unit_type, parent_org_unit_id, head_of_unit, level. Required depth: Financial Entity → Business Area → Department.

1LoD/2LoD Designation

process_id, lod1_owner_user_id, lod1_owner_email, lod2_reviewer_user_id, lod2_reviewer_email. Required for ALL processes in scope — without this the workflow engine cannot route tasks.

User Directory (Azure AD)

user_id, display_name, email, department, business_area, lod_role (1LoD/2LoD/3LoD/ReadOnly/Admin), manager_user_id, active. Synced via Azure AD Graph API.

Reporting Calendar (18 months forward)

event_type, business_area, process_id, due_date, responsible_party, regulatory_basis. Powers the RCSA timeline view and due-date monitor alerts.

5.2 Design Principles
01
Data Sovereignty
All compute and storage within client's Azure tenant in EU region. No data transmitted to external APIs. Azure OpenAI within the client tenant is the only LLM option.
02
Separation of Concerns
Ingestion layer decoupled from storage, decoupled from AI layer. Each layer can be updated independently without impacting other layers.
03
Event-Driven Architecture
TPRM signals and RCSA status changes trigger downstream processing without polling. Azure Service Bus decouples producers and consumers.
04
Defence in Depth
Private Endpoints for all Azure services, RBAC on every resource, audit logging of all AI agent actions, NSGs block all inbound except APIM.
05
Incremental Delivery
Static dashboard first (Phase 1), live data (Phase 2), AI components layered on top (Phase 3). Value is delivered before all AI is ready.

TPRM Signal Engine
Trigger: Azure Service Bus (TPRM event feed) Output: Draft risk for 2LoD review

When vendor classification changes to Critical, auto-generates a draft risk for 2LoD review. Flow: Rule-based pre-filter → Fetch vendor context → Deduplication check → LLM risk drafting → Parse structured JSON → Persist signal → Push notification.

class TPRMSignalEngine: def process_event(self, event: VendorChangeEvent) -> TPRMSignal: # 1. Rule-based pre-filter (classification-relevant event?) # 2. Fetch vendor context, affected processes, SLA breach history # 3. Deduplication check (signal already exists for vendor/field?) # 4. LLM classification and risk drafting (structured JSON output) # 5. Parse and validate against Pydantic model # 6. Persist signal (status: Pending) for 2LoD review # 7. Push notification to relevant 2LoD users
Trend Detection Algorithm
Schedule: Azure Function weekly (Sunday 03:00 UTC) Algorithm: scipy.stats.linregress Min data points: 3 quarters

Identifies processes where residual risk is trending upward over time. Classification: Increasing (slope > 0.5, p < 0.1), Decreasing (slope < -0.5, p < 0.1), or Stable.

def detect_trends(risk_id: str, lookback_quarters: int = 4) -> TrendResult: history = risk_rating_history_repo.get(risk_id, lookback_quarters) if len(history) < 3: return TrendResult(risk_id=risk_id, trend='Insufficient data') scores = [h.residual_likelihood * h.residual_impact for h in history] slope, _, _, p_value, _ = linregress(range(len(scores)), scores) if p_value > 0.1: trend = 'Stable' elif slope > 0.5: trend = 'Increasing' elif slope < -0.5: trend = 'Decreasing' else: trend = 'Stable' return TrendResult(risk_id=risk_id, trend=trend, slope=slope)
RCSA Due-Date Monitor
Schedule: Azure Function daily (07:00 UTC)

Identifies approaching and overdue RCSA assessments and triggers notifications. Logic: (1) Flag overdue assessments → update status to Overdue → alert 2LoD. (2) Find assessments due within 14 days → send reminder if no reminder sent in last 7 days → log reminder to prevent duplication.

1LoD Workflow Engine (Azure Durable Functions)
Platform: Azure Durable Functions v4 Pattern: Human interaction pattern

Manages lifecycle of tasks assigned to 1LoD process owners. RCSA lifecycle states:

Not Started → D-14 Reminder → D-3 Escalation → In Progress → 1LoD Complete → 2LoD Review → Completed
Natural Language Query Interface
Mode 1: Text-to-SQL (data queries) Mode 2: Semantic Search (policy questions)

SQL safety guardrails: read-only database user (ics_agent_readonly); no DML/DDL statements allowed; single statement only; maximum 1,000 rows returned; 10-second query timeout. All queries logged with user_id and timestamp for audit.

5.6 Security Architecture

Network Design

All Azure services behind Private Endpoints. VNet with 4 subnets: app (Container Apps), data (PostgreSQL/Redis), AI (Azure OpenAI/AI Search), integration (APIM/Service Bus). NSGs block all inbound except HTTPS from APIM. No public IP on backend services.

LLM Prompt Security

  • PII minimisation: user_id in prompts; display names injected only when necessary
  • Prompt injection prevention: user input as quoted literal, never concatenated into system prompt
  • Context bounding: LLM accesses only data the requesting user is authorised to see
  • Azure OpenAI config: opt-out-of-training=true; content filtering enabled; EU region only

SSE Reconnection & Fallback

EventSource auto-reconnects on disconnect with exponential backoff (1s, 2s, 4s, 8s, max 30s). If SSE connection cannot be established after 3 retries, fall back to polling GET /api/overview every 60 seconds. A "Live" / "Polling" indicator in the topbar shows the current connection mode to the user.
